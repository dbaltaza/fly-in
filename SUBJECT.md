## Chapter VI — Map File Format

```
nb_drones: 5
start_hub: hub 0 0 [color=green]
end_hub: goal 10 10 [color=yellow]
hub: roof1 3 4 [zone=restricted color=red]
hub: roof2 6 2 [zone=normal color=blue]
hub: corridorA 4 3 [zone=priority color=green max_drones=2]
hub: tunnelB 7 4 [zone=normal color=red]
hub: obstacleX 5 5 [zone=blocked color=gray]
connection: hub-roof1
connection: hub-corridorA
connection: roof1-roof2
connection: roof2-goal
connection: corridorA-tunnelB [max_link_capacity=2]
connection: tunnelB-goal
```

- First line: `nb_drones: <number>`
- Zone types: `start_hub`, `end_hub`, `hub`
- All metadata is optional, enclosed in `[...]`, tags can appear in any order:
  - `zone=<type>` (default: `normal`)
  - `color=<value>` (default: none) — any single-word string
  - `max_drones=<number>` (default: 1)
- Zone types:
  - `normal` — 1 turn cost (default)
  - `blocked` — inaccessible, cannot be entered
  - `restricted` — 2 turn cost
  - `priority` — 1 turn cost, but must be preferred in pathfinding
- Connections: `connection: <name1>-<name2> [metadata]` — bidirectional; zone names cannot contain dashes
  - `max_link_capacity=<number>` (default: 1)
- Comments start with `#` and are ignored.
- Zone coordinates are always integers. There is always exactly one start and one end zone.

## Chapter VII — Mandatory Part

### VII.1 Pathfinding and Algorithm Requirements

- Drones may move simultaneously. The algorithm must schedule paths to maximize throughput.
- Must handle:
  - Distribution of drones across multiple paths
  - Strategic waiting when movement is not possible
  - Avoidance of path conflicts and deadlocks
- Must account for: path lengths, zone type costs, turn scheduling, graph structure, zone and connection capacity constraints.
- **Visual representation is mandatory** — either:
  - Colored terminal output showing drone movements and zone states
  - A graphical interface displaying the network and drone positions
  - Both (recommended)

### VII.2 Zone Occupancy Rules

- Default max occupancy: **1 drone per zone per turn**.
- `max_drones=N` allows up to N drones simultaneously.
- **Start zone**: unlimited — all drones begin here.
- **End zone**: unlimited — delivered drones accumulate here.
- Two drones may not enter the same zone on the same turn unless capacity allows.
- `max_link_capacity` limits how many drones can traverse the same connection simultaneously.
- Drones may move simultaneously as long as all capacity constraints are respected.

### VII.3 Movement and Turn Mechanics

At each turn, every drone may:
- Move to an adjacent connected zone (if capacity allows).
- Move to a connection toward a restricted zone (2-turn move) — **must** arrive next turn, cannot wait on the connection.
- Stay in place.

Rules:
- Drones moving **out** of a zone free up capacity **that same turn**.
- A zone must have available capacity after all departing drones free their slots.
- For restricted zone transit: drone occupies the connection during transit and MUST arrive after 2 turns.

Movement costs (based on destination zone type):
| Type | Cost |
|------|------|
| `normal` | 1 turn |
| `restricted` | 2 turns |
| `priority` | 1 turn (preferred) |
| `blocked` | inaccessible |

### VII.4 Parser Constraints

- First line must be `nb_drones: <positive_integer>`.
- Program must handle any number of drones.
- Exactly one `start_hub` and one `end_hub`.
- Each zone must have a unique name and valid integer coordinates.
- Zone names: no dashes, no spaces.
- Connections must reference previously defined zones.
- Duplicate connections (`a-b` and `b-a`) are errors.
- All metadata must be syntactically valid.
- Zone types must be one of: `normal`, `blocked`, `restricted`, `priority`.
- Capacity values must be positive integers.
- Any parse error: stop and print a clear message with line number and cause.

### VII.5 Simulation Output Format

- One line per turn.
- Each line lists all drone movements, space-separated: `D<ID>-<zone>` or `D<ID>-<connection>` for restricted transit.
- Drones that don't move are omitted.
- Drones that reach the end zone are no longer tracked.
- Simulation ends when all drones have reached the end zone.

Example:
```
D1-roof1 D2-corridorA
D1-roof2 D2-tunnelB
D1-goal D2-goal
```

### VII.6 Scoring System

- Score = total simulation turns (fewer = better).
- A valid simulation must: comply with all movement/occupancy rules, handle zone type costs, respect all capacity constraints, avoid all conflicts.

Optional secondary metrics:
- Drones moved per turn (throughput efficiency)
- Average turns per drone
- Total weighted path cost
- Quality of visual representation

### VII.7 Performance Benchmarks

General targets:
- Easy maps: < 10 turns
- Medium maps: 10–30 turns
- Hard maps: < 60 turns
- Challenger map: beat the reference record of 45 turns *(optional, does not affect grade)*

Reference targets per map:

| Map | Drones | Target |
|-----|--------|--------|
| Easy: linear path | 2 | ≤ 6 turns |
| Easy: simple fork | 4 | ≤ 8 turns |
| Easy: basic capacity | 4 | ≤ 6 turns |
| Medium: dead end trap | 5 | ≤ 12 turns |
| Medium: circular loop | 6 | ≤ 15 turns |
| Medium: priority puzzle | 5 | ≤ 12 turns |
| Hard: maze nightmare | 8 | ≤ 30 turns |
| Hard: capacity hell | 12 | ≤ 35 turns |
| Hard: ultimate challenge | 15 | ≤ 45 turns |
| Challenger: impossible dream | 25 | Reference: 45 turns |
